# 第02章：データと型（変数・キャスト・参照型）

プログラミングでは、データを保存する箱のことを **「変数（へんすう）」** と呼びます。

Javaは、この箱の種類（ **型** ）にとても厳しい言語です。

## はじめに：なぜ「型」を宣言するのか？（Why）

「型を書かなくてもよい言語」は存在します（Python・JavaScript など）。
なぜ Java はわざわざ型を書かせるのでしょうか？

**答え: コンパイラが「間違いを早期に発見」してくれるからです。**

たとえば「年齢」を格納する変数に `"hello"` という文字を入れようとしたとき、
型があれば Java はその場でエラーを出して止めてくれます。

```java
int age = "hello"; // コンパイルエラー！ → バグを実行前に発見できる
```

「型の宣言が面倒」と感じるでしょうが、これは **後で困らないための事前チェック** です。

## 1. プログラミングとは？（心構え）

プログラミングとは、コンピュータへの「お願い（命令）」を書くことです。

ただし、コンピュータは人間のように「空気を読む」ことができません。

**一言一句、正確に命令しないと動きません。**

* **= は「等しい」ではない**

    プログラミングで使う **=** は、数学の「イコール」とは全く別物である。

    「代入（右のものを左に入れる）」という意味である。

* **エラーは友達**

    赤い文字のエラーが出ても、落ち込む必要はない。

    「ここが分からないよ」というコンピュータからの返事である。

## 2. 「型」ってなに？

「型（Type）」とは、**データの形** のことです。

変数は「箱」ですが、Javaでは入れるものに合わせて、箱の形を最初に決める必要があります。

| 型 | 意味 | 例 |
| :--- | :--- | :--- |
| `int` | 整数 | `10`, `-5`, `0` |
| `double` | 小数 | `3.14`, `0.05` |
| `char` | 1文字だけ | `'A'`, `'あ'`（シングルクォート） |
| `String` | 文字列 | `"Hello"`, `"Java"`（ダブルクォート） |
| `boolean` | 真偽値 | `true`（〇）, `false`（×） |

違う形の箱に無理やりデータを詰め込もうとすると、Javaはエラーを出して止めてくれます。

> **補足:** Java 10 以降では、型の代わりに `var` と書くとコンパイラが型を自動で推測してくれる機能があります。
> この講座では型をしっかり身につけるため、`var` は使わずに型を明示して書きます。
>
> **[Java 7 との違い]** `var` は Java 10 以降の機能です。Java 7 以前のコードベースでは使えません。
> 現場の既存コードに `var` が出てきたら、Java 10 以降で書かれたコードだと判断できます。

### 変数の「宣言」と「代入」は分けて書ける

変数は「宣言（箱を用意する）」と「代入（値を入れる）」をまとめて書くのが一般的ですが、分けて書くこともできます。

```java
int score;       // ① 宣言（箱を用意する）
score = 100;     // ② 代入（箱にデータを入れる）
```

ただし、宣言だけして何も入れないまま使おうとするとコンパイルエラーになります。

```java
int score;
System.out.println(score); // コンパイルエラー: variable score might not have been initialized
```

現場では「宣言と同時に代入する」書き方が読みやすいため推奨されます。

### その他のプリミティブ型

上の表以外にも、扱うデータの大きさ（メモリサイズ）によって使い分けられる型があります。
[Oracle公式ドキュメント(dev.java)](https://dev.java/learn/language-basics/primitive-types/) も参考にしてください。

| 型の名前 | サイズ（ビット） | 最小値 / 最大値 | 値の例 | 特徴 |
| :--- | :--- | :--- | :--- | :--- |
| **byte** | 8ビット | -128 ～ 127 | `100` | メモリ節約が必要な大きな配列等に有効。 |
| **short** | 16ビット | -32,768 ～ 32,767 | `10000` | byteと同様、メモリ効率を重視する場合に使用。 |
| **int** | 32ビット | -2³¹ ～ 2³¹-1 | `100000` | 整数を扱う際の標準的な型。 |
| **long** | 64ビット | -2⁶³ ～ 2⁶³-1 | `123456L` | intを超える広範囲の数値に使用。末尾に`L`が必要。 |
| **float** | 32ビット | IEEE 754単精度浮動小数点 | `3.14f` | 浮動小数点の配列でメモリを節約する場合に。末尾に`f`が必要。 |
| **double** | 64ビット | IEEE 754倍精度浮動小数点 | `123.4` | 小数点を含む数値の標準的な型。 |
| **boolean** | （定義なし） | true / false | `true` | 真偽値（フラグ）を保持。サイズはVM依存。 |
| **char** | 16ビット | \u0000 (0) ～ \uffff (65,535) | `'A'` | 単一のUnicode文字。シングルクォートで記述。 |

> **Note**
> `Unicode文字`というのは世界中のすべての文字（漢字、ひらがな、アルファベット、絵文字など）に、重複しない一意の番号（背番号）を割り振った世界共通の規格です。
> 鋭い人なら、ここで疑問が出てきます。なぜ16ビットで絵文字を含めた世界の文字列が扱えるのでしょうか？気になった人は調べてみてください。
> 文字数をカウントするということの難しさが分かってくるだろう。

## 3. 計算とキャスト（Calc.java）

**Calc.java** を実行して、計算のルールを確認しましょう。

### 割り算の罠

**5 / 2** の結果が **2.5** ではなく **2** になりませんでしたか？

* **原因:** 「整数(int)」÷「整数(int)」の答えは、Javaでは「整数（あまり切り捨て）」になるルールだからである。
* **解決策:** **(double)** という「キャスト（型変換）」を使って、「一時的に実数として計算して！」と指示できる。

```java
(double) 5 / 2  →  2.5 になる！
```

### 型の不一致エラーと解決

Javaでは、「大きなデータ(doubleなど)」を「小さな箱(int)」に入れようとするとエラーになります。

```java
int result = price * tax; // エラー！ (1000 * 0.1 = 100.0 は double型)
```

`(int)` というキャストを使って、「小数点以下が消えてもいいから、無理やり整数に変換して！」と命令できます。

```java
int result = (int)(price * tax);
//           ^^^^  ^^^^^^^^^^^^
//           キャスト  計算式を () でまとめている
```

`(int)` は型変換の命令で、その直後の `(price * tax)` は「計算してから変換する」ためにカッコでまとめています。

## 4. よくある間違い（アンチパターン）

### ミス1: int に小数を代入しようとする

整数型（`int`）の箱に小数を直接入れようとするとエラーになります。

```java
// NG: int は小数を持てない
int price = 9.99; // コンパイルエラー: incompatible types

// OK: 小数には double を使う
double price = 9.99;
```

### ミス2: キャストは「四捨五入」ではなく「切り捨て」

`(int)` キャストは小数点以下を**無条件に切り捨て**ます。四捨五入ではありません。

```java
System.out.println((int) 1.9);  // 出力: 2 ではなく 1
System.out.println((int) -1.9); // 出力: -2 ではなく -1
```

四捨五入が必要な場面では `Math.round()` を使います（詳細は第05章）。

### ミス3: char（1文字）と String（文字列）を混同する

```java
// NG: char はシングルクォート。ダブルクォートで囲むと String になる。
char initial = "J"; // コンパイルエラー: incompatible types

// OK: char はシングルクォート、String はダブルクォート
char initial = 'J';   // 1文字。シングルクォート。
String name = "Java"; // 文字列。ダブルクォート。
```

## 5. プリミティブ型と参照型（ReferenceDemo.java）

Javaのデータには、プリミティブ型以外に **「参照型（さんしょうがた）」** があります。

代表的なのが、文字列を扱う **String** です。

| 特徴 | プリミティブ型 | 参照型 |
| --- | --- | --- |
| 名前 | 小文字 (`int`, `char` など) | 大文字 (`String` など) |
| 中身 | データそのもの | データの「場所情報」 |
| 機能 | なし（ただの数値） | 便利な機能（メソッド）を持っている |

### 参照型を使ってみよう

**ReferenceDemo.java** を実行してみましょう。 `String` 型の変数だけ、 `.length()`（長さを測る）などの便利な機能を持っていることを確認できます。

#### 覚え方

* `char`（プリミティブ）: `'A'`（シングルクォート）。ただの1文字データ。
* `String`（参照型）: `"ABC"`（ダブルクォート）。高機能なオブジェクト。文字の並び。

### 参照型の比較には `.equals()` を使う

`String` を比較するとき、`==` ではなく `.equals()` を使います。

```java
// ========== Before: == で比較してしまうアンチパターン ==========
// == は「データの場所（アドレス）」を比較するため、期待通りにならないことがある
String a = "Java";
String b = "Java";
System.out.println(a == b);      // true になることもあるが、信頼できない

// ========== After: .equals() で比較する ==========
// .equals() は「中身のデータ」を比較する。String の比較はこちらを使う。
System.out.println(a.equals(b)); // 常に true（中身が同じなので）
```

> **現場での注意**: `==` で String を比較してしまうバグは現場でも非常によく起こります。`String の比較には .equals()` を使う癖を今からつけておきましょう。

#### なぜ `==` が `true` になることがあるのか

`ReferenceDemo.java` を実行すると `==` の結果も `true` と表示されます。これはバグではありません。

Java は文字列リテラル（`"Hello"` のようにダブルクォートで書いた文字列）をメモリ上で使い回す「**文字列定数プール**」という仕組みを持っています。
そのため `greeting1 = "Hello"` と `greeting2 = "Hello"` は、たまたま同じ場所を指しており `==` が `true` になります。

しかし `new String("Hello")` で作ると、プールを使わず新しい場所にデータが作られるため `false` になります。

```java
String greeting1 = "Hello";
String greeting2 = "Hello";
System.out.println(greeting1 == greeting2);  // true（文字列定数プールで同じ場所を指す）

String greeting3 = new String("Hello");
System.out.println(greeting1 == greeting3);  // false（new で別の場所に作られた）
```

`==` の結果は「**どう作ったか**」によって変わる不安定な比較です。中身を比べたいときは必ず `.equals()` を使いましょう。

## 第02章のまとめ

* **型**: 箱の形。データに合わせて `int` や `double` などを使い分ける。
* **キャスト**: 型を変換すること。`(int)` や `(double)` と書いて強制的に変換する。キャストは四捨五入ではなく切り捨て。
* **変数の更新**: `number = number + 1` は「numberに1を足して、numberに入れ直す」という意味。
* **参照型の比較**: `String` の比較には `==` ではなく `.equals()` を使う。

## コラム：命名のルールとよく使う英語

プログラムで名前をつけるとき、Java には決まった書き方（命名規則）があります。チームの誰が読んでも理解しやすいコードにするための共通ルールです。

### 1. 命名規則

| 種類 | 規則 | 例 |
| --- | --- | --- |
| 変数・メソッド | camelCase（小文字始まり、単語の区切りを大文字に） | `playerName`, `totalScore`, `getAge()` |
| クラス | PascalCase（大文字始まり、単語の区切りを大文字に） | `Player`, `BankAccount`, `ArrayLimitation` |
| 定数 | SCREAMING\_SNAKE\_CASE（全大文字、単語の区切りをアンダースコアに） | `MAX_SIZE`, `DEFAULT_TIMEOUT` |
| パッケージ | すべて小文字 | `com.example.introduction` |

> 定数は `static final` で宣言した変数です。詳しくは第06章で学びます。

### 2. よく使う英語一覧

変数名・メソッド名に使われる英語パターンです。覚えておくと命名に迷わなくなります。

#### 基本操作

| **set** | 値をセット・更新する。 | `setPassword(pw)` |

| **delete** | データそのものを抹消・物理削除する。 | `deleteUser()` |
| **clear** | 中身をすべて空にする。 | `clearAll()` |

### 3. 命名のコツ

#### ① 具体的な名前をつける（抽象的すぎない）

意味のない名前や、短すぎる名前は避けましょう。

* **NG:** `data`, `info`, `a`, `list1`
* **OK:** `userData`, `productPriceList`, `targetDate`
* **例外:** `for` 文のカウンタ変数（`i`, `j`）などは短くてOKです。

#### ② ローマ字・日本語英語を避ける

開発現場はグローバルな基準に合わせるのが一般的です。

* **NG:** `namae`, `kingaku`, `shohinList`
* **OK:** `name`, `amount`, `productList`

#### ③ メソッド名：[動詞] ＋ [目的語/補語]

メソッドは「動作」を表すため、**必ず動詞から始める**のがJavaの鉄則です。

* **動詞 ＋ 名詞（目的語）**：最も一般的な形
  * `sendEmail()` / `saveOrder()` / `updatePassword()`
* **動詞 ＋ 形容詞（補語）**：状態を変化させる場合
  * `makeVisible()`（可視化する） / `keepAlive()`（生存状態を維持する）
* **動詞 ＋ 副詞**：動作の仕方を補足する場合
  * `saveAsync()`（非同期で保存する） / `findAll()`（すべて見つける）

#### ④ 変数名：[形容詞/修飾語] ＋ [名詞]

変数は「データ（もの）」を表すため、**名詞を最後に置く**と中身が理解しやすくなります。

* **形容詞 ＋ 名詞**
  * `currentUser`（現在のユーザー） / `maxRetryCount`（最大リトライ回数）
* **名詞 ＋ 名詞**（前の名詞が後ろを修飾する）
  * `orderId`（注文ID） / `productPrice`（商品の価格）

#### ⑤ boolean型の変数・メソッド：[助動詞/be動詞] ＋ [形容詞/過去分詞]

真偽値は「～か？」という質問形式にするのがJavaの標準です。

* **is ＋ 形容詞**：状態を問う
  * `isEnabled`（有効か？） / `isEmpty`（空か？）
* **is ＋ 過去分詞**：受動的な状態を問う
  * `isAuthorized`（認可されているか？） / `isDeleted`（削除されたか？）
* **has ＋ 過去分詞/名詞**：完了や所有を問う
  * `hasFinished`（完了したか？） / `hasPermission`（権限を持っているか？）

### ⚠️ 注意したい「逆転」のパターン

初心者がやってしまいがちな、不自然な語順の例です。

| NG例（不自然） | OK例（Java標準） | 理由 |
| :--- | :--- | :--- |
| `nameGet()` | `getName()` | 動作（動詞）を先に書くのがJavaのルール。 |
| `countMax` | `maxCount` | 英語では形容詞（max）が名詞（count）の前に来る。 |
| `validIs` | `isValid` | 質問文にするため、be動詞（is）を先頭に置く。 |

### 💡 現場でのアドバイス：A-B-C 構造

複雑な名前をつけるときは、**[範囲/修飾] - [対象] - [動作]** の順で考えると整理しやすくなります。

* 例：`admin` (範囲) + `User` (対象) + `Delete` (動作) + `Task` (本質)
    → `adminUserDeleteTask`
