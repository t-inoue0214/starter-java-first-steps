# 第04章：クラスとオブジェクト

プログラムに「データをまとめる力」と「不正なデータを防ぐ力」を与えましょう。

## はじめに：なぜ「クラス」が必要なのか？（Why）

犬の情報を管理するプログラムを想像してください。

クラスがない場合、こう書くしかありません。

```java
String dog1Name = "ポチ";
int    dog1Age  = 3;
String dog2Name = "ハチ";
int    dog2Age  = 5;
// 犬が100匹いたら…変数が200個必要！
```

クラスを使えば「犬の設計図」をひとつ作っておくだけで、何匹でも量産できます。
これが **オブジェクト指向** の基本的な考え方です。

---

## 1. クラスとオブジェクトの関係

Javaのプログラムは、主に2つの役割に分かれます。

| 用語 | 役割 | 例え（たい焼き） | コードの例 |
| :--- | :--- | :--- | :--- |
| **クラス (Class)** | **設計図** | たい焼きの「金型」 | `Dog` クラス（Main.java に定義） |
| **オブジェクト (Object)** | **実体** | 焼かれた「たい焼き」 | `new Dog("ハチ", 5)` で作られたもの |

### なぜ分けるの？

「金型」がひとつあれば、「たい焼き」を何個でも量産できるからです。
プログラムでも、**「Dog」という設計図をひとつ作っておけば、「ポチ」や「ハチ」といった別々の犬を量産できます。**

---

## 2. クラスの書き方（Main.java）

[`Main.java`](Main.java) を開いてください。この章では、**比較のために2種類の「犬の設計図」** をひとつのファイルに書いています。

- `DogBefore` クラス：問題のある設計（Before）
- `Dog` クラス：改善した設計（After）

クラス（設計図）には、主に2つのことを書きます。

### フィールド (Field)

そのモノが持つ「データ」。名前や年齢など。

```java
String name;
int age;
```

**用語の整理:**

| 用語 | 説明 |
| :--- | :--- |
| **ローカル変数** | メソッドの中で作る変数。そのメソッドが終わると消える |
| **フィールド** | クラスの中で作る変数。オブジェクトが生きている間ずっと保持するデータ |
| **メンバ変数** | 「フィールド」と同じ意味。古い本やC++経験者がこう呼ぶことがある |

### メソッド (Method)

そのモノができる「動作」。吠える、自己紹介するなど。

```java
void bark() { ... }
```

**「void」ってなに？:**

メソッド名の前にある **void**（ボイド）は、「空っぽ」という意味です。
「このメソッドを実行しても、結果（データ）は返しません。やりっぱなしでOKです」という印です。

結果を返してほしい場合は、ここに返してほしい型（`int` や `String`）を書きます。詳しくは「4. After」で見てみましょう。

---

## 3. Before：public フィールドの問題点（DogBefore）

[`Main.java`](Main.java) の `DogBefore` クラスを見てください。フィールドが `public` になっています。

```java
public class DogBefore {
    public String name; // 外部から自由に書き換えられてしまう
    public int age;
}
```

このクラスを使うと、こんな問題が起きます。

```java
DogBefore dog1 = new DogBefore();
dog1.name = "ポチ";
dog1.age  = 3;
dog1.introduce(); // 正しく表示される

// 問題: 不正な値を何でも書き込める！
dog1.age = -100; // -100歳の犬は存在しないのに、エラーにならない
dog1.introduce(); // -100歳の犬が表示されてしまう！
```

**なぜこれが「問題」なのか？**

銀行口座の残高が `public` だったとしたら、誰でも残高を `-9999999` に書き換えられる、ということです。
データを「外から自由に触れる状態」にしておくのは、現場では危険な設計です。

---

## 4. After：private + コンストラクタ + ゲッター（Dog）

### アクセス修飾子：public と private の違い

| キーワード | 意味 | 使いどころ |
| :--- | :--- | :--- |
| `public` | **どこからでも** アクセスできる | 外部に公開したいメソッドに使う |
| `private` | **そのクラスの中だけ** でアクセスできる | 外から触られたくないフィールドに使う |

> **[補足]** アクセス修飾子には他に `protected`（サブクラスからアクセス可）とパッケージプライベート（修飾子なし）もありますが、これらは継承と合わせて第06章で学びます。

`Dog` クラスではフィールドを `private` にしています。

```java
private String name; // このクラスの外からは直接触れない
private int age;
```

これで `dog2.age = -100;` と書くとコンパイルエラーになります。
Javaがプログラムを実行するより先に、間違いを発見してくれます。

### コンストラクタ

`new Dog("ハチ", 5)` のように書くと、オブジェクトを作ると同時に値を設定できます。
この「オブジェクト生成時に必ず呼ばれる特別なメソッド」を **コンストラクタ** と呼びます。

```java
public Dog(String name, int age) {
    this.name = name;
    this.age  = age;
}
```

コンストラクタには2つのルールがあります。

1. **クラス名と同じ名前**にする（`Dog` クラスなら `Dog`）
2. **戻り値の型を書かない**（`void` もつけない）

### this キーワード

`this.name = name;` の **this** は **「このオブジェクト自身」** を指します。

コンストラクタの引数名（`name`）とフィールド名（`name`）が同じで紛らわしいため、`this.` をつけてフィールドを区別しています。

```java
this.name = name;
//  ↑フィールドの name   ↑引数の name
```

### ゲッター

フィールドが `private` になると、外部から値を読み取ることもできなくなります。
そこで「**読み取り専用の窓口**」として **ゲッター (Getter)** を用意します。

```java
public int getAge() {
    return age; // 読み取りはできるが、書き換えはできない
}
```

ゲッターがあれば `dog2.getAge()` で年齢を取得できます。
しかし `dog2.age = -100;` は依然としてコンパイルエラーになります。「読める・でも書けない」という状態です。

> [!TIP]
> 「後から値を変えたい場合は `setAge()` を書けばいい」と思った人は鋭い。
> setter をいつ書くべきか、なぜ `Dog` クラスに setter がないのかは、第08章のイミュータブル設計で詳しく扱います。

### カプセル化とは

`private` フィールド + コンストラクタ + ゲッターの組み合わせで、データを外部の不正な操作から守ることを **カプセル化** といいます。
カプセル（薬のカプセル）のように、中身を外から直接触れない形で包む、というイメージです。

---

## 5. よくある間違い（アンチパターン）

### NG例1：コンストラクタに void を書いてしまう

```java
// NG: void をつけるとコンストラクタではなくメソッドとして扱われてしまう
public void Dog(String name, int age) { ... }

// OK: 戻り値の型は書かない
public Dog(String name, int age) { ... }
```

`void` をつけると「`Dog` という名前のメソッド」として扱われてしまいます。
コンストラクタは「戻り値の型を書かない」のがルールです。

### NG例2：コンストラクタの中で this を忘れる

```java
// NG: 引数の name に引数の name を代入しているだけで、フィールドが更新されない！
public Dog(String name, int age) {
    name = name;
    age  = age;
}

// OK:
public Dog(String name, int age) {
    this.name = name;
    this.age  = age;
}
```

`this.` を書かないと「引数の `name` に引数の `name` を代入する」という意味になり、
フィールドが空のまま（`null` や `0`）になります。

### NG例3：フィールドを何でも public にしてしまう

```java
public int age;  // NG: 外部から -100 などの不正な値を書き込まれる
private int age; // OK: カプセル化でデータを守る
```

「動けばいい」という考えで全フィールドを `public` にするのは、現場では避けるべき設計です。

---

## 第04章のまとめ

- **クラス**: データ（フィールド）と動作（メソッド）をひとまとめにした「設計図」
- **オブジェクト**: `new` キーワードで設計図から生成した「実体」。それぞれ独立したデータを持つ
- **アクセス修飾子**: `public`（どこからでもOK）/ `private`（クラス内のみ）でアクセスを制限する
- **コンストラクタ**: オブジェクト生成時に呼ばれる初期化処理。クラス名と同じ名前・戻り値の型なし
- **this**: そのオブジェクト自身を指すキーワード。フィールドと引数が同名のときに区別する
- **ゲッター**: `private` フィールドを「読み取り専用」で公開するメソッド
- **カプセル化**: `private` + コンストラクタ + ゲッターでデータを守る設計の考え方
