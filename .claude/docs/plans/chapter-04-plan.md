# 第04章 実装計画: class_and_objects（クラスとオブジェクト）

## 概要

プログラムに「データをまとめる力」と「不正なデータを防ぐ力」を与えるためのオブジェクト指向の基礎を学ぶ。
初心者にとって最大のハードルとなる「クラス」や関連用語の概念を、身近な例え（たい焼き・犬）を用いて分かりやすく整理し、なぜオブジェクト指向（カプセル化）が必要なのかを Before/After の対比で体験させる。

## 必須実装内容

### 1. クラスと関連用語の概念整理

初心者が混乱しやすい用語を、役割と例えで明確に区別して説明する。

- **クラス**: 自分で作れる新しい「型」であり、データと動作をまとめた「設計図」（例：犬の設計図、たい焼きの金型）
  - プリミティブ型（シンプルな箱）に対する、自分で自由に作れる「複雑な箱（新しい型）」として説明する。
- **オブジェクト（インスタンス）**: 設計図から作られた「実体」（例：ポチ、焼かれたたい焼き）
- **メソッド**: オブジェクトができる「動作・ふるまい」。
- **コンストラクタ**: オブジェクトが誕生する瞬間に呼ばれる「初期設定係」。

#### 【設計意図】手続き型の限界を体験してから OOP を導入する

本章は「なぜクラスが必要なのか」という Why を体験で納得させることが核心となる。
そのため、以下の構成を意図的に採用する。

- 第01〜03章では「処理を上から順に並べる手続き型の書き方」を体験してきた。
- 管理する変数が増えるにつれて（例：「名前・年齢・種族・色・体重」をすべてバラバラな変数で管理する）、手続き型では限界が来ることを Before として示す。
- その限界を解決する手段として「データと振る舞いをひとまとめにする OOP の発想」を After として導入する。
- これが「なぜクラスが必要なのか」という Why の核心となり、この Before/After の落差をできる限り大きく体験させる構成にする。

### 2. Java 標準の参照型との関係

- Java 標準の参照型（`String`、`ArrayList`、`Scanner` など）は、あらかじめ用意された「公式のクラス」に過ぎない。
- 自分が作るクラス（`Dog`、`User` など）も同じ仕組みで動く「オリジナルのクラス」であることを示し、自分でも便利な部品を作れることを実感させる。

### 3. クラスの役割（データを保持するクラス）

本章では「データを保持するクラス」の設計に集中する。

- **データを保持するクラス**の例え: 「社員の履歴書」「犬のプロフィール」
- 特徴: フィールド（データ）と、それに対する簡単な操作（ゲッター・コンストラクタ）だけを持つ。
- 現状の `Main.java` は `DogBefore`・`Dog` というデータ保持クラスで構成されており、この方針に沿って実装済みである。

「処理を行うクラス」（`Calculator` など、自分のデータはあまり持たずにデータ保持クラスを受け取って処理する役割）の分離は、第14章（設計とアーキテクチャ）で扱う。本章では実装しない。

### 4. カプセル化の実装と意義（Why の理解）

- **public フィールドの問題点**: 外部から不正な値を自由に書き込まれる危険性（銀行口座の残高の例など）。
- **アクセス修飾子**: `public`（公開）と `private`（非公開）の使い分け。
- **カプセル化の実現**: `private` フィールド + コンストラクタ + ゲッターを組み合わせ、データを外部の不正な操作から守る実装手法。
- **this キーワード**: オブジェクト自身を指し、フィールドと引数を区別する使い方。

#### 【設計意図】第04章では setter を意図的に実装しない

本章（第04章）では setter（`setXxx()`）を `Dog` クラスに一切実装しない。これは見落としではなく意図的な設計判断となる。

- 第04章の学習目標は「`private` フィールド + コンストラクタ + ゲッターでデータを守る」という基本形の習得であり、「ゲッターを使う = カプセル化」という理解で十分とする。
- setter の是非（mutable クラスが引き起こす問題）に踏み込むのは第08章（`ImmutableDesign.java`）の役割とする。
- 学習者が第04章の時点で「setter を書くべきか」という疑問を持った場合は「第08章で詳しく扱う」と一言コメントを添えるにとどめる。
- 「第04章の `Dog` クラスに setter を書かなかった理由の答え」は第08章の `ImmutableDesign.java` で明示的に明かされる設計になっている（章をまたいだ伏線）。

## アンチパターン

現状の `class_and_objects/README.md` に掲載済みのアンチパターンを記録する。

### NG例1：コンストラクタに void を書いてしまう

```java
// NG: void をつけるとコンストラクタではなくメソッドとして扱われてしまう
public void Dog(String name, int age) { ... }

// OK: 戻り値の型は書かない
public Dog(String name, int age) { ... }
```

`void` をつけると「`Dog` という名前のメソッド」として扱われてしまう。コンストラクタは「戻り値の型を書かない」のがルール。

### NG例2：コンストラクタの中で this を忘れる

```java
// NG: 引数の name に引数の name を代入しているだけで、フィールドが更新されない！
public Dog(String name, int age) {
    name = name;
    age  = age;
}

// OK:
public Dog(String name, int age) {
    this.name = name;
    this.age  = age;
}
```

`this.` を書かないと「引数の `name` に引数の `name` を代入する」という意味になり、フィールドが空のまま（`null` や `0`）になる。

### NG例3：フィールドを何でも public にしてしまう

```java
public int age;  // NG: 外部から -100 などの不正な値を書き込まれる
private int age; // OK: カプセル化でデータを守る
```

「動けばいい」という考えで全フィールドを `public` にするのは、現場では避けるべき設計。

## 注意点

### アクセス修飾子の範囲について

`public` と `private` の使い分けは第04章の主担当とする。

`protected` とパッケージプライベート（アクセス修飾子なし）は継承と密接に関連するため、第06章の継承・インターフェース説明の文脈で触れることとし、第04章では扱わない。

### 学習者への配慮

- クラスやオブジェクトの説明には「たい焼き」や「犬」といった直感的に理解しやすい例えを一貫して使う。
- Before（問題のあるコード）と After（カプセル化されたコード）を比較させることで、「なぜその仕組みが必要なのか（Why）」を体験で納得させる構成にする。
- `ArrayList<String>` の `<String>` の意味は「詳しくは第06章で学ぶ」という一行注記にとどめ、深入りしない。

### 既存コンテンツとの整合性

- 第03章（制御フロー）で学んだ `if-else` / `for` / `while` は、本章の `main` メソッド内で当然のように使う（前章の知識を使うことで定着を促す）。
- 第05章では `new Scanner()`・`new ArrayList()` という「Java 標準クラスのインスタンス化」を扱う。本章で `new Dog()` を学ぶことが、その前提知識になる。
