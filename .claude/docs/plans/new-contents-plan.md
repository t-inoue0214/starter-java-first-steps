## 要望1: 理解度チェック問題 — 推奨方針と詳細計画

### 推奨方針: 各章末尾に統一的に設ける

**全体まとめ章にする案を採用しない理由:**

- このリポジトリは「各章でWhyを体験する」構成を取っている。章末問題は「この章で体験したことを自分の言葉で確認する」という役割を持つ。チャプター横断の総復習は知識が遠い章との記憶の干渉を生みやすく、初心者にとって負担が大きい。
- ch01〜05ですでに各章末スタイルが確立されており、学習者はそのパターンに慣れている。全体まとめ章を追加すると「第15章」が増えるか、既存章末から問題を移す大規模改修が必要になる。
- ch08以降でも同じパターンで設計すれば一貫性が保たれる。

**各章末統一スタイルに決める理由:**

- 「この章の内容だけで完結する問題」として設計でき、学習者が直前の学習内容で答えられる。
- ch01〜05の既存パターンを踏襲するため追加コストが低い。
- ch06・ch07の「次章への問いかけ」セクションを「確認してみよう」に統合・整理するだけで完成する。

## 要望2: 命名ルール + よく使う英語一覧 — 推奨方針と詳細計画

### 選択肢の比較

| 配置案 | メリット | デメリット |
| --- | --- | --- |
| ch01 README 冒頭に追加 | 最初から参照できる | 第01章は「環境確認・Hello World」に集中する章。ルール一覧を冒頭に置くと情報過多になり初心者が混乱する |
| ch02 README 内部に追加（型説明の直後） | 変数の命名と型宣言をセットで学べる | ch02はすでに型・キャスト・参照型の3テーマで情報量が多い。命名ルールを差し込む余地が薄い |
| 独立リファレンスファイル（各章の参照元） | どこからでも参照できる。章の文章量が増えない | リポジトリ構成が例外的になる（章フォルダ以外にファイルが生まれる）。学習フローに組み込まれないため読まれない可能性がある |
| appendix章（第00章または付録フォルダ） | 体系的にまとめられる | 章番号の命名規則（数字で始められない）との整合性問題が生じる。追加コストが大きい |
| **ch02 README 末尾（「確認してみよう」直前）に専用セクションとして追加** | 変数名を初めて書く直前のタイミングで参照できる。既存章への追加で済む。Javaの識別子ルールと型の章で命名を扱うのは自然 | ch02の文章量がやや増える |

### 推奨方針: ch02 README 末尾（「確認してみよう」直前）に専用セクションを追加

**採用理由:**

- 学習者が初めて変数名を自分でつける実習（`Calc.java` の改変）を行うのがch02であるため、命名ルールを学ぶ最適なタイミングがch02の学習中となる。
- ch04（クラス名）・ch06（インターフェース名・定数）への自然な橋渡しになる。各章で「この命名ルールを思い出してください」と参照するだけで済む。
- 独立ファイルよりも学習フローに組み込まれるため、読まれる確率が上がる。

### 追加する内容

#### セクション名: 「コラム: 命名のルールとよく使う英語」

ch02 README の「第02章のまとめ」と「確認してみよう」の間に挿入する。

##### 命名規則の表

| 対象 | 記法 | 例 |
| --- | --- | --- |
| 変数名・メソッド名 | camelCase（小文字始まり） | `userName`, `getAge`, `isLoggedIn` |
| クラス名・インターフェース名 | PascalCase（大文字始まり） | `BankAccount`, `Comparable` |
| 定数（`static final`） | SCREAMING_SNAKE_CASE（大文字＋アンダーバー） | `MAX_SIZE`, `DEFAULT_TIMEOUT` |
| パッケージ名 | 全小文字（数字・アンダーバーは使わない） | `com.example.introduction` |

##### よく使う英語動詞・接頭辞の表

| 英語 | 意味 | 使う場面の例 |
| --- | --- | --- |
| `get` | 取得する | `getName()`, `getAge()` — フィールドの値を返すゲッター |
| `set` | 設定する | `setName("Alice")` — フィールドに値を入れるセッター |
| `is` / `has` | 〜かどうか（真偽値） | `isEmpty()`, `hasNext()` — boolean を返すメソッド |
| `add` / `put` | 追加する・入れる | `add("item")`, `put("key", value)` — コレクションへの追加 |
| `remove` / `delete` | 削除する | `remove(0)`, `deleteUser(id)` — 要素の削除 |
| `find` / `search` | 探す | `findById(1)`, `searchByName("Alice")` — 検索系 |
| `create` / `make` / `build` | 作る | `createOrder()`, `buildUrl()` — 生成系 |
| `update` / `modify` | 更新する | `updateStatus()`, `modifyRecord()` — 更新系 |
| `load` / `fetch` | 読み込む・取ってくる | `loadConfig()`, `fetchData()` — 外部からデータを取得 |
| `save` / `store` | 保存する | `saveToFile()`, `storeResult()` — 永続化系 |
| `validate` / `check` | 確認・検証する | `validateInput()`, `checkExists()` — バリデーション系 |
| `convert` / `parse` | 変換する・解析する | `convertToInt()`, `parseJson()` — 型変換・フォーマット変換 |
| `init` / `initialize` | 初期化する | `initDatabase()`, `initialize()` — 最初の設定 |
| `count` / `size` | 数を数える | `countItems()`, `size()` — 件数を返す |
| `MAX` / `MIN` / `DEFAULT` | 最大・最小・既定値 | `MAX_RETRY`, `DEFAULT_TIMEOUT` — 定数の接頭辞 |

##### 注釈（学習の流れとの接続）

- クラス名（PascalCase）の実例は第04章で `Dog` クラスを設計するときに出てくる。
- 定数（SCREAMING_SNAKE_CASE）の実例は第06章の `enum` と `static final` で出てくる。

## 注意点

### 学習者への配慮

- ch06・ch07 の「確認してみよう」問題は、**各章のファイルを実行した後に初めて意味を持つ設計**にする。問題文に「XxxJava を実行した後で」という前提を明示する。
- コード修正型の問題は「元に戻してください」を必ず添える。特に ch06 の `@Override` 誤字問題は、誤字のままにしておくとコンパイルが通らなくなる。

### 命名コラムの範囲

- ch02は超入門レベルの章であるため、命名コラムはあくまで「参照用」として提示する。「テストに出る」という圧迫感を与えないよう、セクション名を「コラム:」としてoptionalな読み物であることを示す。
- `SCREAMING_SNAKE_CASE` と `static final` の関係はこの時点ではまだ習っていないため、「詳しくは第06章で学ぶ」という一言を添えるにとどめる。

### 既存コンテンツとの整合性

- ch06 の「次章への問いかけ」セクションは **削除せずに残す**。「確認してみよう」が「この章で学んだことを確認する問題」であるのに対し、「次章への問いかけ」は「次の章への知的な引き込み」という異なる役割を持つため。
- ch07 の「次章（第08章）への問いかけ」も同様に残す。
- ch08〜14 の実装計画ファイル（`chapter-XX-plan.md`）には、章実装時に「確認してみよう」セクションの設計を必ず含めることをルールとして追記することを推奨する（本計画書のスコープ外）。
