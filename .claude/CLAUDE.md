# CLAUDE.md

## プロジェクト概要

新卒プログラマを中級プログラマへ引き上げるJava学習リポジトリ。変数・四則演算を知った初心者が、制御構文からモダンJava・設計・並行処理まで段階的に学ぶ。

- **対象:** Java入門者（変数・四則演算は理解済み）→ 中級プログラマ
- **環境:** GitHub Codespaces / OpenJDK 21 / VS Code
- **重視する思想:** 「なぜ現場でその書き方をするのか（Why）」を体験で学ぶ

## 開発ワークフロー

作業は必ず以下のステップで進めること：

1. **探索 (Explore):** 既存コードや README を `grep` / `read` で調査し、現状を把握する。
2. **計画 (Plan):** 修正方針を `docs/plans/` 配下に Markdown で作成し、ユーザーの承認を得る。
   - **自律修正:** `@education-specialist`がレビューを行い、教育的観点からの改善（優先度高・中）は、エージェントが自律的に連続して修正・検証を繰り返すことを許可する。
3. **実装 (Implement):** 承認された計画に基づき、`@java-programmer`（コード）または `@education-specialist`（文書）が作業を行う。
   - **自律修正:** 教育的観点からの改善（優先度高・中）は、エージェントが自律的に連続して修正・検証を繰り返すことを許可する。
4. **検証 (Verify):** `javac` でコンパイルし、`java` で実行結果が意図通りか確認する。

## 重要ルール

- フォルダ名 = パッケージ名（数字で始められないため番号プレフィックスなし）
- Why ヘッダーコメント（`/** 【なぜこのコードを学ぶのか】 */`）を全 `.java` ファイルに必ず付ける
- Before/After の対比で「問題を先に体験させてから解決策を示す」構成にする
- コメントは日本語・Java 21 モダン構文（`->`, `var` 等）を積極的に使う
- Java 8 以降で追加された構文・API には **Java SE 7 互換性注釈** を付ける

## Markdown 編集ルール

- 編集後は markdownlint-cli2 と textlint を実行し、エラーがゼロになるまで修正する
- markdownlint-cli2 と textlint の指摘事項の修正はユーザーの承認を得ず、自動的に修正を行う
- コマンドが見つからない場合は処理を中断してユーザーに確認する
- mermaid記法をサポートしているため、第08章以降では必要に応じてシーケンス図やクラス図を少し記述する
